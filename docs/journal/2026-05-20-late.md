## Critical bug: model learned `<reply>` → `<eos>`

When sampling against the midsize checkpoint, almost every response was empty. Diagnostic on logits at the prompt `<user> Hello there.\n<reply>`:

```
top tokens after <reply>:
  id=    2 score=  21.82 tok='<eos>'
  id=    1 score=  14.58 tok='<bos>'
  ...
```

The model learned that `<reply>` is almost always followed by `<eos>`. Root cause: `scripts/pack_shards.py` appended `<eos>` after **every** blank-line-separated paragraph block. In dialogue corpora like the Gutenberg-extracted chunks, those paragraphs are 32-utterance dialogue blocks, but each block ended with a `<reply>` line — so the model's training data taught it `<reply>` precedes `<eos>` overwhelmingly.

## Fix

`scripts/pack_shards.py` updated: only insert `<eos>` after accumulating ≥ 600 characters of source text. Short dialogue chunks stay glued so the model sees extended conversation flow rather than constant turn-then-end patterns. With this change, re-tokenize + re-pack + re-train.

## What the result is *actually* good for

Even with the `<eos>`-bias bug, the run validated three real things:

1. **Sustained training throughput on the 5070 for a 50M model at 1024 seq len is ~72k tok/s.**
   - Linear scaling estimate to the full 350M target at 2048 seq len: ~12k tok/s.
   - 7B tokens at 12k tok/s = ~580k sec ≈ 7 days continuous GPU burn for the full Chinchilla-optimal pretrain. Matches the 1–4 week range I estimated earlier.

2. **The Phase 0 → Phase 0.5 jump is real.**
   - 12M params → loss 4.0
   - 50M params → loss 3.1
   - Same data, 4× params, ~0.9 loss improvement. Curve scaling is roughly tracking what you'd expect.

3. **bf16 + AdamW + the whole stack is stable** across 6000+ steps without divergence, gradient explosions, or numeric issues. Solid foundation for the long run.

## Eval results on midsize (all near-zero due to the bug)

- **Sycophancy probe:** 0 markers / 39 prompts. Mostly empty responses, so this is a trivially-zero result, not a real "Glemton beats RLHF'd models" win. The eval *pipeline* worked end-to-end; the *result* doesn't yet say anything about the angle.
